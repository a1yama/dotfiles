## レビュー対象

以下のコマンドで変更差分を取得し、レビューしてください。

```bash
git diff
git diff --cached
```

未追跡の新規ファイルがある場合は、ファイル内容を直接読んでレビューしてください。

```bash
git status --short
```

## レビュー観点

1. **バグ・ロジックエラー** — 意図しない動作、エッジケースの見落とし、off-by-one エラー
2. **セキュリティ** — インジェクション、ハードコードされた秘密情報、安全でない入力処理
3. **可読性・保守性** — 不明瞭な命名、不必要な複雑さ、過剰な抽象化
4. **一貫性** — 既存コードベースのスタイル・パターンとの整合性

## 言語固有のチェックポイント

変更されたファイルの言語に応じて、`languages/` ディレクトリ内の該当ファイルを参照してください:
- Go → `languages/go.md`
- TypeScript/JavaScript → `languages/typescript.md`
- PHP → `languages/php.md`
- Python → `languages/python.md`
- Rust → `languages/rust.md`

各ファイルには、言語特有のよくあるミスと参考資料がまとめられています。

## 出力フォーマット

問題がある場合のみ報告してください。問題がなければ「問題なし」と報告してください。

各指摘は以下の形式で簡潔に:

```
[重要度: high/medium/low] ファイルパス:行番号
内容: 問題の説明
提案: 修正案
```

## レビューループ

報告の末尾に、必ず次のどちらかを明記してください。

- 指摘あり: 「**呼び出し元へ: 指摘を修正した後、code-review を再実行してください。`問題なし` になるまで繰り返します（最大3回。超えたら残課題を列挙してユーザーに判断を仰ぐ）。**」
- 指摘なし: 「問題なし」

## チェックリスト追加候補

レビュー中に `languages/<言語>.md` に未掲載の「再発しそうなミスのパターン」を見つけた場合は、該当言語のチェックリストを読んで重複がないことを確認した上で、報告の末尾に以下の節を含めてください。

```
### チェックリスト追加候補
- 言語: <言語> / カテゴリ: <カテゴリ> / 概要: <一行> / 説明: <詳細>
```

**呼び出し元へ: この節が報告に含まれていた場合は、update-review-checklist スキルを候補の内容で実行し、チェックリストに反映してください（ユーザーへの確認は不要）。**
