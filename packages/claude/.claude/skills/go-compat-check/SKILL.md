ビルドは通るが実行時に壊れる「サイレントな破壊的変更」を、ライブラリの実装そのものを読んで検出する。CHANGELOGやリリースノートには頼らない。認知外の問題を見つけることが目的であり、ライブラリのソースコードと自コードを突合して判定する。

## なぜこのスキルが必要か

ライブラリのバージョンアップでビルドが通っても、実行時にpanicしたり挙動が静かに変わることがある。CHANGELOGを読んでも人間は見落とす。テストがない箇所では検出手段がなく、インシデント���直結する。

実例: go-playground/validator v10.15.1→v10.20.0で、`dive`タグがスライス/マップ以外のフィールドに使われるとpanicするようになった。ビルドは通り、該当コードが実行されるまで誰も気づかなかった。

## 変更ライブラリの特定

以下の順序でgo.modの差分を探す。最初に差分が見つかった時点で次に進む。

1. `git diff go.mod` — ステージング前の変更（go mod tidy直後の典型パターン）
2. `git diff --cached go.mod` — ステージング済みの変更
3. 両方で差分がなければ「変更されたライブラリはありません」と報告して終了する

HEAD~1やそれ以前のコミットにフォールバックしてはならない。��ーザーが明示的にdiff範囲を指定した場合のみ、その指定に従う。

差分からバージョンが変更された依存ライブラリ（indirect含む）を全て抽出する。

## 自コードの使用箇所を洗い出す

変更された各ライブラリについて、自コード（vendor/やtestdata/を除く `*.go`）からimportしている箇所をGrepで探す。見つかったファイルを読み、以下の使用パターンを抽出する:

- 関数・メソッド呼び出し
- 構造体タグ（validate, gorm, json, db等）
- 型アサーション・型スイッチ
- インターフェース実装
- 定数・変数の参照

## ライブラリ実装の読み込みと突合

`go env GOMODCACHE` でモジュールキャッシュのパスを取得し、`$GOMODCACHE/<module>@<version>/` 配下のソースを読む。見つか��なければ `go mod download` してから再試行する。

自コードが���っているAPIの実装だけに絞って読む。ライブラリ全体を読む必要はない。

### 何を探すか

根本的な考え方: **ライブラリが期待する使い方と、自コードの実際の使い方にギャップがないか**を探す。以下は代表的なパターンだが、これに限定せず異常を感じたら報告する。

- **panic条件**: reflect.Kindのチェックやswitch文での型分岐でpanicに到達する条件。diveタグの例はまさにこれ
- **構造体タグの制約**: タグの値とフィールド型の組み合わせが、ライブラリのタグ処理ロジックで許容されるか
- **nil/空値の扱い**: メソッドの戻り値やポインタ引数への書き込みで、nil��空スライスのどちらが返るか。呼び出し側のnilチェックと整合するか
- **デフォルト値の変更**: 未設定時の挙動が変わっていないか
- **非推奨API**: `// Deprecated:` コメントがある関数を自コードが使っていないか

### 分析のコツ

- reflect.Kindのチェック、switch文での型分岐は重点的に読む。ここにpanic条件が潜んでいることが多い
- panic()やlog.Fatal()のコードパスに到達する条件を逆算して洗い出す
- 検出した問題がバージョンアップで新たに導入されたものか、旧バージョンにも存在する既存バグかを区別する。既存バグでも実行時に壊れるならcriticalとして報告する（ただし「既存バグ」と補足する）

## レ���ート出力

リポジトリルートに `go-compat-report-YYYY-MM-DD.md` として出力する。

レポートの構成:

1. **ヘッダ**: 実行日、diff範囲、検査ライブラリ数、検出件数（critical/warning/info）
2. **Critical / Warning / Info**: 各問題につ��て、ファイルパス・行番号、自コードでの使い方、ライブラリ実装の該当箇所、リスク（何が起きるか）、推奨対応を記載
3. **検査済みライブラリ一覧**: ライブラリ名、旧→新バージョン、使用箇所数、検出数のテーブル
4. **バージョン間の主な変更点**: 自コードに影響し��る変更の箇条書き（破壊的変更だけでなく、バリデーション厳格化等の挙動変更も含��）

### severity判定

- **critical**: 実行時panic、データ破損、セキュリティ問題に直結する確実性の高いもの
- **warning**: 挙動変更により期待と異なる結果になる可能性があるもの
- **info**: 非推奨APIの使用、将来のバージョンで問題になりうるもの

確実なものだけをcriticalにする。推測レベルはwarning/infoとし、断定できない場合はその旨を正直に書く。ライブラリのソースが読めない場合はスキップしてレポートに記載す���。
